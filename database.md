# OxideRelay Database Schema

OxideRelay uses SQLite as the only database for MVP.

The database stores:

* Users
* Permissions
* Projects
* Project access
* Languages
* Environments
* Namespaces
* Translation keys
* Translation values
* Sessions

PostgreSQL is out of scope for MVP.

Environment access control in MVP is permission-based only.

There is no `user_environment_access` table.

For project-scoped authorization, project ownership is evaluated before checking direct permission assignments.

The project owner is treated as being able to perform any action inside the owned project through implicit project-scoped and environment-scoped permissions.

---

# General Rules

## IDs

Use UUID strings for primary keys.

```sql
id TEXT PRIMARY KEY
```

UUIDs are generated by the application.

---

## Timestamps

Use ISO-8601 strings in UTC.

```sql
created_at TEXT NOT NULL
updated_at TEXT NOT NULL
```

For immutable join tables, `updated_at` is not required.

---

## Soft Delete

Do not implement soft delete in MVP.

Use physical delete where needed.

---

# Tables

## users

Stores system users.

```sql
CREATE TABLE users (
    id TEXT PRIMARY KEY,
    email TEXT NOT NULL UNIQUE,
    password_hash TEXT NOT NULL,
    display_name TEXT NOT NULL,
    is_active INTEGER NOT NULL DEFAULT 1,
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL
);
```

Indexes:

```sql
CREATE INDEX idx_users_email ON users(email);
```

---

## permissions

Permissions are atomic access rights.

```sql
CREATE TABLE permissions (
    id TEXT PRIMARY KEY,
    code TEXT NOT NULL UNIQUE,
    description TEXT
);
```

Examples:

```text
ManageUsers
ManagePermissions

CreateProjects
EditProjects
DeleteProjects
ViewProjects
ManageProjectMembers

ReadTranslations
EditTranslations
DeleteTranslations
ImportTranslations
ExportTranslations

ReadDevelopment
ReadStaging
ReadProduction

EditDevelopment
EditStaging
EditProduction
```

The permission catalog is seeded and immutable in MVP.

`ManagePermissions` allows assigning and removing direct user permissions from users.

It does not allow runtime creation of new permission codes.

---

## user_permissions

Direct user permissions.

Used for exceptions and fine-grained access.

```sql
CREATE TABLE user_permissions (
    user_id TEXT NOT NULL,
    permission_id TEXT NOT NULL,

    PRIMARY KEY (user_id, permission_id),

    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
    FOREIGN KEY (permission_id) REFERENCES permissions(id) ON DELETE CASCADE
);
```

Indexes:

```sql
CREATE INDEX idx_user_permissions_permission_id
ON user_permissions(permission_id);
```

---

## projects

Stores localization projects.

The user who creates a project becomes its owner.

In MVP, the project owner implicitly has all project-scoped permissions inside that project without requiring direct global permission assignment.

```sql
CREATE TABLE projects (
    id TEXT PRIMARY KEY,
    name TEXT NOT NULL,
    slug TEXT NOT NULL UNIQUE,
    description TEXT,
    owner_user_id TEXT NOT NULL,
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL,

    FOREIGN KEY (owner_user_id) REFERENCES users(id) ON DELETE RESTRICT
);
```

Indexes:

```sql
CREATE INDEX idx_projects_slug ON projects(slug);
CREATE INDEX idx_projects_owner_user_id ON projects(owner_user_id);
```

---

## user_project_access

Defines which projects a user can access.

```sql
CREATE TABLE user_project_access (
    user_id TEXT NOT NULL,
    project_id TEXT NOT NULL,
    created_at TEXT NOT NULL,

    PRIMARY KEY (user_id, project_id),

    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
    FOREIGN KEY (project_id) REFERENCES projects(id) ON DELETE CASCADE
);
```

Indexes:

```sql
CREATE INDEX idx_user_project_access_project_id
ON user_project_access(project_id);
```

## languages

Languages enabled for a project.

```sql
CREATE TABLE languages (
    id TEXT PRIMARY KEY,
    project_id TEXT NOT NULL,
    code TEXT NOT NULL,
    name TEXT NOT NULL,
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL,

    FOREIGN KEY (project_id) REFERENCES projects(id) ON DELETE CASCADE,

    UNIQUE (project_id, code)
);
```

Examples:

```text
en
ru
sr
de
```

Indexes:

```sql
CREATE INDEX idx_languages_project_id
ON languages(project_id);
```

## environments

Translation environments for a project.

```sql
CREATE TABLE environments (
    id TEXT PRIMARY KEY,
    project_id TEXT NOT NULL,
    name TEXT NOT NULL,
    slug TEXT NOT NULL,
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL,

    FOREIGN KEY (project_id) REFERENCES projects(id) ON DELETE CASCADE,

    UNIQUE (project_id, slug)
);
```

Default environments:

```text
development
staging
production
```

Indexes:

```sql
CREATE INDEX idx_environments_project_id
ON environments(project_id);
```

## namespaces

Logical groups of translation keys.

```sql
CREATE TABLE namespaces (
    id TEXT PRIMARY KEY,
    project_id TEXT NOT NULL,
    name TEXT NOT NULL,
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL,

    FOREIGN KEY (project_id) REFERENCES projects(id) ON DELETE CASCADE,

    UNIQUE (project_id, name)
);
```

Examples:

```text
common
validation
checkout
profile
```

Indexes:

```sql
CREATE INDEX idx_namespaces_project_id
ON namespaces(project_id);
```

---

## translation_keys

Stores translation keys.

```sql
CREATE TABLE translation_keys (
    id TEXT PRIMARY KEY,
    project_id TEXT NOT NULL,
    namespace_id TEXT NOT NULL,
    key TEXT NOT NULL,
    description TEXT,
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL,

    FOREIGN KEY (project_id) REFERENCES projects(id) ON DELETE CASCADE,
    FOREIGN KEY (namespace_id) REFERENCES namespaces(id) ON DELETE CASCADE,

    UNIQUE (project_id, namespace_id, key)
);
```

Examples:

```text
button.save
button.cancel
required
```

Indexes:

```sql
CREATE INDEX idx_translation_keys_project_id
ON translation_keys(project_id);

CREATE INDEX idx_translation_keys_namespace_id
ON translation_keys(namespace_id);
```

---

## translation_values

Stores actual translated values.

```sql
CREATE TABLE translation_values (
    id TEXT PRIMARY KEY,
    translation_key_id TEXT NOT NULL,
    language_id TEXT NOT NULL,
    environment_id TEXT NOT NULL,
    value TEXT NOT NULL,
    updated_by_user_id TEXT,
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL,

    FOREIGN KEY (translation_key_id) REFERENCES translation_keys(id) ON DELETE CASCADE,
    FOREIGN KEY (language_id) REFERENCES languages(id) ON DELETE CASCADE,
    FOREIGN KEY (environment_id) REFERENCES environments(id) ON DELETE CASCADE,
    FOREIGN KEY (updated_by_user_id) REFERENCES users(id) ON DELETE SET NULL,

    UNIQUE (translation_key_id, language_id, environment_id)
);
```

`id` is the row primary key.

This `id` is the identifier used by translation update and delete API routes.

The composite unique constraint prevents duplicate values for the same translation key, language, and environment.

Indexes:

```sql
CREATE INDEX idx_translation_values_key_id
ON translation_values(translation_key_id);

CREATE INDEX idx_translation_values_language_id
ON translation_values(language_id);

CREATE INDEX idx_translation_values_environment_id
ON translation_values(environment_id);

CREATE INDEX idx_translation_values_lookup
ON translation_values(language_id, environment_id);
```

---

## sessions

Stores active admin UI sessions.

```sql
CREATE TABLE sessions (
    id TEXT PRIMARY KEY,
    user_id TEXT NOT NULL,
    expires_at TEXT NOT NULL,

    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE

```

Indexes:

```sql

ON sessions(user_id);

```

# Initial Administrator

On first startup:

1. Check if any user exists.
2. If no users exist, create an initial administrator user.
3. Use environment variables:

```text
OXIDERELAY_ADMIN_EMAIL
OXIDERELAY_ADMIN_PASSWORD
```

4. Assign all seeded permissions directly to that user.

These variables are required only during the bootstrap case when no users exist yet.

If at least one user already exists, startup must not require them.

---

# Default Project Initialization

When a project is created:

1. Create the project.
2. Set creator as `owner_user_id`.
3. Add creator to `user_project_access`.
4. Create default environments:

```text
development
staging
production
```

5. Create default namespace:

```text
common
```

6. Treat the project owner as having all project-scoped and environment-scoped permissions inside that project.

No per-project environment ACL records are created because environment access is determined by permissions plus the implicit owner rule.

Project creation must be executed in a single database transaction together with owner access creation, default environments, and default namespace creation.

# Query Examples

## Get translations for static JSON endpoint

Endpoint:

```http
GET /static/{project_slug}/{environment_slug}/{language_code}/{namespace}.json
```

SQL logic:

```sql
SELECT
    tk.key,
    tv.value
FROM translation_values tv
JOIN translation_keys tk ON tk.id = tv.translation_key_id
JOIN languages l ON l.id = tv.language_id
JOIN environments e ON e.id = tv.environment_id
JOIN namespaces n ON n.id = tk.namespace_id
JOIN projects p ON p.id = tk.project_id
WHERE p.slug = ?
  AND e.slug = ?
  AND l.code = ?
  AND n.name = ?
ORDER BY tk.key;
```

Response:

```json
{
  "button.save": "Save",
  "button.cancel": "Cancel"
}
```

---

## Get all translations for backend API

Endpoint:

```http
GET /api/v1/projects/{project_slug}/locales/{language_code}?environment=production
```

SQL logic:

```sql
SELECT
    n.name AS namespace,
    tk.key,
    tv.value
FROM translation_values tv
JOIN translation_keys tk ON tk.id = tv.translation_key_id
JOIN languages l ON l.id = tv.language_id
JOIN environments e ON e.id = tv.environment_id
JOIN namespaces n ON n.id = tk.namespace_id
JOIN projects p ON p.id = tk.project_id
WHERE p.slug = ?
  AND e.slug = ?
  AND l.code = ?
ORDER BY n.name, tk.key;
```

Response format:

```json
{
  "project": "hr-portal",
  "locale": "ru",
  "environment": "production",
  "values": {
    "common.button.save": "Сохранить",
    "common.button.cancel": "Отмена",
    "validation.required": "Required field"
  }
}
```

Key composition rule:

```text
translation_keys.key stores only the local key part.
The delivery response key is composed as {namespace}.{key}.
```

# Import Behavior

Import target must include:

```text
project_id
environment_id
language_id
namespace_id
```

Suggested admin endpoint:

```http
POST /api/v1/projects/{project_slug}/imports/json
```

Required permissions:

```text
ImportTranslations
Edit{Environment}
```

Input JSON:

```json
{
  "button.save": "Save",
  "button.cancel": "Cancel"
}
```

For each entry:

1. Find or create `translation_keys`.
2. Upsert `translation_values`.
3. Update `updated_by_user_id`.
4. Update `updated_at`.

Input keys are local to the selected namespace and must not include the namespace prefix.

---
